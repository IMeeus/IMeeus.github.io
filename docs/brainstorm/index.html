Brainstorm

Use of setups

  • why
    • less code / duplication per test,
    • bring out more intent
    • removes things we're not interested in.

What to test:

  • user interaction/behavior
    • why
      • gives the developer confidence that a feature works
      • prevents regression of behavior
    • how
      • write tests at the level where everything is in scope to test the behavior.
  • tdd hooks & helpers
    • why
      • hooks & helpers should be pure functions that can be tested in isolation.
      • it is good practice to apply wishfull thinking here.
    • how
      • you all know how...
  • render regression
    • why
      • if you make changes, you want to be notified whether it impacts your render
    • how
      • you don't want to have to re-test a render update in multiple components.
      • for third-party dependency (api / local storage) write your different render paths in the file that does the call.
      • for behavior (button click) write your test at the level where you have all render changes & the button in scope.

What not to do:

  • test propagations
  • mock internal things
    • only mock external: third party / browser storage / api calls / other mfe's
    • don't mock hooks, components, ...
    • if you don't need to mock it, you don't have to...
