User interaction tests

What?​

According to RTL: "The more your tests resemble the way your software is used, the more confidence they can give you". In other words, you should test user interaction and mock as little as possible. Therefore these tests belong to the integration test category.

Examples:

  • It should open the menu, when I click the menu button.
  • It should show a validation error, when I submit an invalid form.

Little history lesson. In 2021 the most popular testing library, enzyme, discontinued to make place for react-testing-library (RTL) as the number 1 library. In contradiction to enzyme, RTL didn't add support for shallow rendering, because the creator had a strong belief that it should never be used.

Why?​

Confidence, Clarity & Regression​

The amount of confidence & clarity of these tests is best explained when you compare them to unit tests.

Unit tests for a single React component would look like this:

  • It should call onClick when the button was clicked.
  • It should show an error, when hasError is true.

A user interaction test would look like this:

  • It should show a validation error, when I submit an invalid form.

Which of these tests would you rather write?

User interaction tests are more pleasant to write, because you're testing something that actually feels important.

Relieves E2E test responsibilities​

As you may know, E2E tests, that spin up your entire application, are costly. So if there's an opportunity that you don't need one, you should take it. Many features in an application are purely frontend, and it's much better to write a low cost user interaction test for these. You should only write an E2E test when a feature has dependencies external to your frontend, such as api-calls to a backend.

How?​

What (not) to mock?​

Never mock components!

In React, behavior is usually spread across multiple components, which are connected by properties, callbacks and contexts. If you mock some of these components, then you also mock their share of behavior, which leads to misleading tests and a false sense of security.

Mocking components also locks you out of using RTL's recommended queries, such as getByRole and getByText.

Only mock API calls!

We are not writing E2E tests. Mock your API calls.

Although, make sure you follow the API mocking guidelines, otherwise you may end up in a "mocking hell".

Where to put these tests?​

Put your behavior test next to the component which manages that behavior.

For example, imagine a page which contains a list of people which can be filtered with a searchbox.

Let's say, we want to write this test:

it("only shows people that I searched for", () => {});

Then the test file for PeopleView would be the correct place to put it, because it has SearchField, SearchButton and PersonList in scope, which are all required to manage which people are shown.

function PeopleView() {
  const allPeople = ["alex", "john"];

  const [shownPeople, setShownPeople] = useState<Person[]>(allPeople);
  const [searchText, setSearchText] = useState<string>();

  const filterPeople = () => {
    setShownPeople(allPeople.filter((p) => p.includes(searchText));
  };

  return (
    <>
      <SearchField onChange={(text) => setSearchText(text)} />
      <SearchButton onClick={() => filterPeople()} />
      <PersonList people={shownPeople} />
    </>
  );
}

Don't query by data-testid! 😣​

The query getByTestId should be a last resort when writing behavior tests.

Quoting @testing-library/react:

The user cannot see (or hear) these, so this is only recommended for cases where you can't match by role or text or it doesn't make sense (e.g. the text is dynamic).

Instead, use getByRole or getByText, as recommended in RTL's priority list.

Follow RTL's recommendations​

There are many more RTL recommendations that you can find in their documentation. Here are a few important ones:

  • Use screen to query DOM elements.
  • Use user-event, instead of fire-event.
